## 背景

原 `anon_insert_submissions` RLS 策略使用 `with check (true)`，允许匿名用户向 `submissions` 表插入任意字段值（包括 `status='approved'`、`approved_tool_slug`、`reviewed_at`、`reject_reason`），可绕过主理人审核流程直接写入已审核状态。

**修复策略：** 将 `with check` 收紧为只允许插入"待审"记录：

```sql
create or replace policy "anon_insert_submissions"
  on public.submissions
  for insert
  to anon
  with check (
    status = 'pending'
    and approved_tool_slug is null
    and reviewed_at is null
    and reject_reason is null
  );
```

---

## 执行过程说明

### 遇到的问题

首次在 Supabase SQL Editor 中仅运行 `CREATE POLICY` 语句（未先执行 `DROP`），返回：

```
ERROR: 42710: policy "anon_insert_submissions" for table "submissions" already exists
```

**原因：** 原迁移 `20260518000001` 已创建同名策略，直接 `CREATE` 会冲突。

**解决方式：** 迁移文件已包含 `drop policy if exists ... ; create policy ...` 两步，需在 SQL Editor 中执行**完整文件**，不能只选中 `create policy` 部分单独运行。

## 测试用例与结果

测试工具：`curl`，角色：`anon`（使用 `NEXT_PUBLIC_SUPABASE_ANON_KEY`）

### 说明：HTTP 401 vs 403

PostgREST 对 `42501`（RLS 违规）的 HTTP 状态码映射：
- **anon 角色**（无 JWT）→ `401 Unauthorized`
- **authenticated 角色**（有 JWT 但无权限）→ `403 Forbidden`

本次使用 anon key 测试，返回 `401` 属于正确行为，语义等同于拦截，下文以 ✅ 标注。
