# Moxie AI 项目聚合平台 — 2 人团队 1 个月 MVP 执行方案

**版本:V2.0 | 日期:2026 年 5 月 13 日**
**适用范围:基于方案 V1.0 技术栈,人力压缩至 2 人,30 天交付可验证 MVP**
**一句话定义:2 人 × 4 周 × AI 辅助开发,跑通"采→洗→展→发→聊"业务闭环,每周一个可演示版本。**

---

## 一、调整背景与核心策略

### 1.1 调整背景

| 项 | 原方案 V1.0 | 本方案 V2.0 |
|---|---|---|
| 人员 | 8 人(后端 ×2 + 前端 ×2 + DevOps + 测试 + PM + 负责人) | **2 人全栈** |
| 周期 | 45 天(6 周) | **30 天(4 周开发 + 1 周冗余)** |
| 数据源 | 5–7 个 | **MVP 期 2 个,Stretch 3 个** |
| 范围 | 数据 + 用户 + 交易 + 后台完整 | **核心闭环 + 极简后台** |
| 测试 | 独立测试工程师 | **开发自测 + 周五交叉验收** |

### 1.2 核心策略(必须严格执行)

1. **AI 是第三名工程师** — 所有重复代码(CRUD/爬虫适配器/表单)由 Claude/Cursor 生成,人 review。预期可节省 40% 编码时间。
2. **范围三档制** — 每周明确 P0(必须)/P1(应该)/P2(可砍)。周中超时立即砍 P2。
3. **第三方零定制** — 环信 IM、阿里云短信、Ant Design Pro 模板**只配置不改源码**。
4. **DevOps 极简** — 单台云主机 + Docker Compose,不上 K8s、不上 ELK、不上 Prometheus。监控用 Uptime Kuma 一个容器搞定。
5. **测试用 AI 生成** — Pytest 用例由 Claude 生成,开发跑自测;周五 Demo = 集成验收。
6. **后台用 Pro 现成页** — 项目审核/数据查看直接用 Ant Design Pro Vue 的 List/Form 模板改字段,不写新页面。

## 二、技术栈(沿用方案 V1.0,无变更)

| 层 | 选型 | 2 人配置下的取舍 |
|---|---|---|
| 爬虫 | Crawl4AI | 保留。每个数据源 ≤ 1 天产出适配器 |
| 数据清洗 | LangChain + Claude 3.5 Sonnet | 保留。MVP 只做"去重 + 字段标准化",不做语义补全 |
| 后端 | FastAPI | 保留 |
| 前端 | Vue 3 + Ant Design Vue 4.x | 保留。**只做 4 个核心页面** |
| 后台 | Ant Design Pro Vue | 保留。**只用 List/Form/Detail 三个模板** |
| 数据库 | PostgreSQL 16 + PgVector | **PgVector 暂不启用**(MVP 不做向量检索),保留扩展能力 |
| 任务队列 | Celery 5.x + Redis 7.x | 保留 |
| IM | 环信 IM 4.x | 保留。**只接 1v1 文字消息** |
| 短信 | 阿里云短信 | 保留 |
| 部署 | Docker Compose 2.x | 保留 |
| 监控 | Uptime Kuma | 保留(1 个容器) |
| 日志 | **文件 + Loki**(可选)/ 起步用本地 docker logs | **砍 ELK**,等上量再补 |
| API 文档 | FastAPI 自动 Swagger | 保留 |

## 三、2 人分工与职责边界

### 3.1 角色定义

| 代号 | 角色 | 核心负责 | 必备技能 |
|---|---|---|---|
| **P1** | 后端主导全栈 | 数据采集、清洗、所有 API、数据库、Celery 任务、Docker 部署、阿里云短信对接 | Python(FastAPI/Crawl4AI/Celery)、PostgreSQL、Linux、Docker |
| **P2** | 前端主导全栈 | 前台 4 页 + 后台 3 页、环信 IM 前端、UI 联调、Nginx 配置、Uptime Kuma 配置 | Vue 3、Ant Design Vue/Pro、JS/TS、基础 Nginx |

### 3.2 严格的职责边界(每日同步同遵守)

| 谁做 | P1 必做 | P2 必做 | 共同负责 |
|---|---|---|---|
| ✅ | 写 SQL / 写 API / 写爬虫 / 写 Celery 任务 / 写 docker-compose.yml | 写 Vue 组件 / 写 CSS / 调 Antd / 集成环信 SDK / 写 nginx.conf | API 契约文档(OpenAPI 自动生成 + Postman 集合)、Bug 互相 review |
| ❌ | 不碰 Vue 代码、不调 CSS | 不写后端业务逻辑、不操作生产数据库 | — |

### 3.3 决策机制

- **API 契约**:P1 周一前定 OpenAPI 草案 → P2 周一站会确认 → 改动需双方同意。
- **范围争议**:超时 / 不确定时,**砍 P2 任务优先**,保 P0/P1 不破线。
- **第三方故障**(短信/IM/Claude API):24h 内评估替代方案,详见 §六. 风险预案。

## 四、MVP 范围(P0/P1/P2 三档)

### 4.1 P0 — 必须完成(决定 MVP 是否成立)

| 模块 | 功能点 | 验收标准 |
|---|---|---|
| 数据采集 | YC 官网爬虫 + GitHub Trending(AI 标签)爬虫 | 每日凌晨 2 点全量,入库率 ≥ 95%,单源数据 ≥ 200 条 |
| 数据清洗 | 基于 URL/GitHub 地址去重 + 标准化字段(名称/简介/标签/链接/采集时间) | 去重准确率 100%,字段非空率 ≥ 90% |
| 定时任务 | Celery Beat 每日凌晨 2:00 触发,失败重试 3 次 | 连续 3 天稳定执行无人工干预 |
| 前台榜单 | 项目列表(分页)+ 按数据源筛选 + 按更新时间排序 | 首屏 ≤ 1.5s,分页正常,适配 PC/H5 |
| 前台详情 | 项目详情页(基础信息 + 跳转原站链接) | 详情字段完整,加载 ≤ 1.5s |
| 用户登录 | 手机号 + 短信验证码登录 + 退出 | 短信 1 分钟内到达,Token 7 天有效 |
| 后台管理 | 项目列表查看 + 手动触发采集任务 + 查看采集日志 | 后台账号密码登录,操作生效 |
| 部署 | Docker Compose 一键起全栈(7 个容器:fastapi/celery-worker/celery-beat/redis/postgres/nginx/uptime-kuma) | 全新 Ubuntu 主机 30 分钟内部署完 |

### 4.2 P1 — 应该完成(完整体验)

| 模块 | 功能点 | 验收标准 |
|---|---|---|
| 数据源扩展 | Product Hunt 爬虫(AI 类目) | 单源数据 ≥ 100 条 |
| 个人中心 | 收藏 / 取消收藏项目 + 我的收藏列表 | 收藏即时生效 |
| 项目发布 | 用户填表发布项目 + 待审核状态 | 表单校验通过即可提交 |
| 后台审核 | 待审核列表 + 通过/驳回(无原因) | 通过后前台可见 |
| 在线咨询 | 项目详情页"咨询"按钮 → 环信 1v1 文字聊天 | 双方收发消息正常 |

### 4.3 P2 — 可砍(时间紧立刻砍)

| 模块 | 功能点 | 砍掉影响 |
|---|---|---|
| Hugging Face 爬虫 | 数据源多样性减弱,不影响闭环 | 低 |
| Crunchbase 爬虫 | 同上,且 Crunchbase 反爬严重,优先砍 | 低 |
| 浏览历史 | 用户可不依赖,收藏已覆盖核心需求 | 低 |
| 数据导出 Excel | 后台手动 SQL 导出可替代 | 低 |
| 智能字段补全 | 用 LLM 补全成本高,MVP 不必做 | 中 |
| 异常数据可视化看板 | 看 Uptime Kuma + docker logs 够用 | 低 |

### 4.4 明确不做(MVP 后再议)

AI 评分、个性化推荐、资金托管、电子协议、APP/小程序、多语言、向量检索、RAG、企业认证、批量审核、消息已读回执、附件/图片消息。

## 九、为什么 2 人 30 天可行(关键支撑论据)

| 节省点 | 怎么节省 | 节省工时 |
|---|---|---|
| 不写爬虫引擎 | Crawl4AI 开箱即用,每个数据源 0.5–1 天写适配器 | 省 10 d |
| 不写后台框架 | Ant Design Pro Vue 模板 + 改字段 | 省 5 d |

| 不写短信 | 阿里云 SDK + 1 个 API | 省 2 d |

**总节省 ≈ 40 人天。** 2 人 × 25 工作日 = 50 人天 + AI 加速约 30% ≈ 65 等效人天 → **覆盖 P0+P1 范围(约 55–60 等效人天)有 5–10 天余量,用 W5 冗余周吸收波动。**

## 十、成功标准(MVP 是否成立的最终判据)

只要满足以下 5 条**全部**为是,即视为 MVP 成功,具备进入下一迭代的资格:

1. **采** — 系统每日自动采集 ≥ 2 个数据源,连续 3 天稳定运行
2. **洗** — 入库数据去重 100%,核心字段非空率 ≥ 90%
3. **展** — 用户在 H5 / PC 看到榜单 + 详情,首屏 ≤ 1.5s
4. **发** — 用户能登录 → 发布项目 → 后台审核 → 前台可见(完整闭环)
5. **聊** — 用户能在项目详情发起 IM 咨询,双方收发正常

未满足任一条,则 W5 冗余周必须补齐;若 W5 仍未达成,则**召开范围裁剪会**,只保留满足的部分上线,其余推到 V1.1。
